# Trade Blotter — Durable Trade History & Query Layer

**Status:** Planned  
**Priority:** High — core infrastructure gap  
**Scope:** persistence.py, lifecycle_engine.py, strategy.py, dashboard.py, deploy scripts

---

## Problem

CryoTrader writes completed trades to `logs/trade_history.jsonl` but never reads them back. The trade history is effectively write-only:

1. **`load_trade_history()` is dead code** — zero callers in the entire codebase
2. **No slot ID recorded** — can't attribute trades to specific slots
3. **No close trigger recorded** — lose the "why" (stop-loss, take-profit, expiry, manual)
4. **No execution context** — `execution_mode`, mark price at close, index price at close are not persisted
5. **No dashboard view** — `dashboard.py` only shows live state; no `/api/history` endpoint
6. **File destroyed on redeploy** — `deploy-slot.sh` rsyncs the working tree and wipes `logs/`. A redeploy on Apr 17 lost trades #1–4.
7. **CLOSED trades accumulate in memory** — `_trades` dict in `LifecycleEngine` never evicts completed trades. `trades_snapshot.json` grows without bound.

The net effect: after a process restart or redeploy, the application has **no memory** of what it did. Stats reset, PnL resets, and the only record is a fragile JSONL file that may or may not still exist.

---

## Design Principles

- **Append-only, immutable records** — once written, a trade record is never modified
- **Durable by default** — trade history survives process restarts AND redeploys
- **Rich context at write time** — capture everything you'll want later, because you can't go back
- **Queryable** — filter by date range, slot, strategy, instrument; powers dashboard and CLI
- **Separated from lifecycle** — the blotter is an independent persistence concern, not coupled to whether the `TradeLifecycle` object is alive in memory
- **No database** — JSONL is fine for our scale (10–20 trades/day). The gap is enrichment and durability, not storage engine.

### Step 1: Enrich the JSONL record

**File:** `persistence.py` → `save_completed_trade()`

Add these fields to the record dict:


**Where these values come from:**
- `SLOT_ID` — already set in every slot's `.env` file (generated by `slot_config.py`); just read it via `os.environ`
- `close_trigger` — already written to `trade.metadata["close_trigger"]` by exit condition logic in `lifecycle_engine.py` (line ~419); just persist it
- `execution_mode` — already a field on `TradeLifecycle` (`trade.execution_mode`)
- `index_price_at_close` / `mark_prices_at_close` — **new**: must be captured in `lifecycle_engine.py` at close time from `self._market_data`; store in `trade.metadata` before `_finalize_close()`

**Backward compatibility:** New fields default to `None` on old records. `load_trade_history()` returns raw dicts — no schema enforcement needed.

### Step 2: Make trade history durable

**Problem:** `logs/trade_history.jsonl` lives inside the working tree. `deploy-slot.sh` rsyncs the local tree to the VPS, which can overwrite or delete the remote `logs/` directory.

**Solution:** Write to a path **outside** the deploy tree on the VPS.

```python
# persistence.py — choose history file location
import os

def _history_file_path() -> str:
    """Durable path that survives redeploys.
    
    On the VPS each slot runs as ~/cryotrader-slot-XX/.
    We write history to ~/trade_history/slot-XX.jsonl instead,
    which is outside the rsync target.
    
    Locally (development), fall back to logs/trade_history.jsonl.
    """
    slot_id = os.environ.get("SLOT_ID")
    durable_dir = os.path.expanduser("~/trade_history")
    if slot_id and os.path.isdir(os.path.dirname(durable_dir)):
        os.makedirs(durable_dir, exist_ok=True)
        return os.path.join(durable_dir, f"slot-{slot_id}.jsonl")
    # Fallback for local dev
    os.makedirs("logs", exist_ok=True)
    return os.path.join("logs", "trade_history.jsonl")
```

Replace the hardcoded `HISTORY_FILE = os.path.join("logs", "trade_history.jsonl")` with a call to `_history_file_path()`.

**Deploy script change:** Add `~/trade_history/` to the VPS setup if it doesn't exist. Do NOT include it in rsync targets.

### Step 3: Wire up `load_trade_history()` with query support

**File:** `persistence.py`

Replace the current bare `load_trade_history()` with a filtered version:

```python
def load_trade_history(
    self,
    slot_id: Optional[str] = None,
    strategy_id: Optional[str] = None,
    since: Optional[float] = None,       # unix timestamp
    until: Optional[float] = None,       # unix timestamp
    symbol_contains: Optional[str] = None,
) -> List[Dict[str, Any]]:
    """Load completed trades with optional filters.
    
    All filters are AND-combined. Returns newest-first.
    """
    history_file = _history_file_path()
    if not os.path.exists(history_file):
        return []

    records = []
    with open(history_file, "r") as f:
        for line in f:
            line = line.strip()
            if not line:
                continue
            record = json.loads(line)
            
            if slot_id and record.get("slot_id") != slot_id:
                continue
            if strategy_id and record.get("strategy_id") != strategy_id:
                continue
            if since and (record.get("closed_at") or 0) < since:
                continue
            if until and (record.get("closed_at") or 0) > until:
                continue
            if symbol_contains:
                legs = record.get("open_legs", [])
                if not any(symbol_contains in l.get("symbol", "") for l in legs):
                    continue
            records.append(record)
    
    records.sort(key=lambda r: r.get("closed_at") or 0, reverse=True)
    return records
```

### Step 4: Dashboard trade history endpoint

```python
@app.route("/api/history")

def api_history():
    """Trade history with optional query params: days, strategy, slot."""
    days = int(request.args.get("days", 7))
```

### Step 5: Evict CLOSED trades from `_trades`

**File:** `lifecycle_engine.py`

Currently, CLOSED and FAILED trades remain in `self._trades` forever. `strategy.py`'s `stats` property relies on `all_trades` to compute win/loss/PnL.

**Change:**
1. After `save_completed_trade()` is called in `strategy.py → _check_closed_trades()`, mark the trade ID for eviction
2. In `LifecycleEngine`, add an `evict_trade(trade_id)` method that removes from `_trades`
3. Move `stats` computation to read from the JSONL blotter (via `load_trade_history()`) instead of in-memory `all_trades`

**Critical dependency:** `stats` must work from the blotter, not from memory, before eviction can be enabled. Otherwise restarting the process would reset stats to zero (which is what happens today anyway — so this is strictly an improvement).

**Phasing:** Implement stats-from-blotter first, then enable eviction. If stats-from-blotter proves complex, defer eviction to a follow-up and just cap `_trades` with a max-age prune (e.g., drop trades closed >24h ago).

### Step 6: Capture market context at close time

**File:** `lifecycle_engine.py`

At close time (just before calling `trade._finalize_close()`), capture:

```python
# In the CLOSING → CLOSED transition, before _finalize_close():
try:
    index_price = self._market_data.get_index_price()
    trade.metadata["index_price_at_close"] = index_price
    # Mark prices for each leg's symbol
    marks = {}
    for leg in trade.open_legs:
        mark = self._market_data.get_mark_price(leg.symbol)
        if mark is not None:
            marks[leg.symbol] = mark
    trade.metadata["mark_prices_at_close"] = marks
except Exception:
    pass  # Best-effort — don't fail the close
```

There are **two code paths** that transition to CLOSED in `lifecycle_engine.py`:
1. Normal close (~line 408–430) — after close legs fill
2. Expiry close (~line 480–500) — when options expire worthless

Both paths must capture this context.

## What NOT to do

- **No database.** JSONL is sufficient for 10–20 trades/day. A database adds operational complexity (migrations, backups, connection management) for no benefit at this scale.
- **No exchange reconciliation.** The Apr 19 latency test proved Deribit returns complete fill data (including fees) in the order response itself. There is no data quality gap to reconcile against.
- **No real-time streaming.** The dashboard polls via htmx. No WebSocket/SSE needed.
- **No trade modification.** The blotter is append-only. If a record is wrong, append a correction note — don't edit history.

## Empirical Context (Apr 19, 2026)

These findings from production inform the design:

### Trade log availability is instant
The Deribit trade log latency test (BTC-26JUN26-75000-C, ATM call) measured:
- Open trade log: **35ms** (1 poll)
- Close trade log: **36ms** (1 poll)
- `_trades` array with fill details and fees was **in the order response itself**

Conclusion: no deferred reconciliation needed. Our inline fee capture via `extract_fee()` (v1.16.0) is authoritative.

### Fee impact is material
From the Apr 13–19 slot-02 analysis (short_strangle_delta_tp on Deribit):
- 5 trades: **0.00751 BTC ($565)** total fees
- Fees consumed **26.5%** of gross profit
- Single-trade PnL overstatement: **10.3%**

Fee-inclusive PnL (Part A, implemented Apr 19) now captures this correctly. The blotter persists the corrected values.

### Redeploy data loss
A redeploy on Apr 17 wiped `trade_history.jsonl`, losing trades #1–4. The durable path design (Step 2) prevents this class of data loss entirely.
